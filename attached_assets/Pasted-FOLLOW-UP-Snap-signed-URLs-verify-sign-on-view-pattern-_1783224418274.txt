FOLLOW-UP: Snap signed URLs — verify sign-on-view pattern, not sign-on-upload
The signed-URL fix works, but I need to confirm WHEN the signing happens. The wrong pattern breaks two cases:

Snaps already in the DB were stored with the broken /object/public/... URL string — if the viewer renders the stored media_url from the row, all existing snaps stay black.
If a signed URL is generated at UPLOAD time and stored in media_url, the URL's 24h TTL starts at upload — a recipient opening a snap 23 hours after it was sent gets a URL that dies in 1 hour, and an unviewed snap's URL can expire before its expires_at.

DIAGNOSE AND REPORT:

Where does createSignedUrl get called — at upload (then stored in media_url) or at view time when the recipient opens the snap?
What exactly is stored in snaps.media_url for a new snap — a full signed URL, or just the storage path?

Store ONLY the storage path (e.g. snaps/{userId}/{filename}) in snaps.media_url (or add a media_path column if changing semantics is cleaner — if a migration is needed, OUTPUT THE SQL FILE ONLY, do not run it).
